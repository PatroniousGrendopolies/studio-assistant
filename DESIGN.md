## Problem Statement

Autoland is a subscription-based recording studio where clients book rooms and use professional audio equipment. Onboarding currently takes 60-120 minutes of staff time per session. Clients range from total beginners (never touched a mic stand) to seasoned engineers. The biggest risks are equipment damage from misuse (heavy mics on light stands, phantom power on ribbon mics, bad cable handling) and client frustration when they can't figure out signal flow without help.

The goal: an AI chat assistant accessible via QR code in each room that cuts onboarding to under 30 minutes, catches dangerous actions before they happen, and lets clients get help without waiting for staff.

## What Makes This Cool

A studio client scans a QR code, says "I want to record vocals with the big tube mic," and the AI knows their room, knows that mic is heavy, asks what stand they're using, and catches them before they put a $5,000 mic on a $30 stand. That's the moment. Not a chatbot that answers FAQs. An assistant that understands analog studio physics and protects expensive gear through conversation.

Image troubleshooting (photo of patchbay, AI sees what's wrong) is deferred to v2. The v1 "whoa" moment is the deterministic safety pre-scan: the server catches dangerous keywords *before* the LLM even sees the message, so safety warnings fire instantly and reliably.

## Constraints

- Mobile-first web app (PWA), no native app, no app store
- No login or accounts. QR scan to session. Zero friction.
- Cannot see or control hardware. All safety guardrails are triggered by what the user says, not by sensors or monitoring.
- Knowledge base must be human-reviewed. The AI cannot invent safety procedures.
- Must work on client phones, laptops, and studio computers.
- Budget-friendly: Vercel free tier + Claude API pay-per-use.
- SOPs must be updatable without code changes (markdown files in `/content`).

## Premises

1. **The knowledge base is the bottleneck, not the AI.** LLMs already know audio production well. The hard part is Autoland-specific room configs, wiring diagrams, equipment inventories, and step-by-step SOPs. These are lighter than a full audio engineering textbook, more like structured equipment lists, signal flow maps, and procedural checklists.

2. **Voice input and the symptom picker are essential for beginners, not nice-to-haves.** A client with no audio vocabulary won't type a useful query. The "I don't know what to say" button and voice input are the primary interface for most users.

3. **Safety guardrails are the #1 value driver.** One prevented equipment damage incident (phantom power on a ribbon mic, tube mic falling off a light stand) pays for the entire project. This drives every priority decision.

4. **No login for MVP.** QR code to session. Adding a login wall kills adoption in a studio where people are there to record, not to create accounts.

5. **The studio works today without this.** Staff onboarding works. This makes it faster and safer, but the MVP can be genuinely minimal because the fallback is "go find a staff member."

## Approaches Considered

### Approach A: "Big Prompt, Small App" (CHOSEN)

Next.js app deployed on Vercel. QR code per room encodes a room ID in the URL. On load, the app reads content files for that room (markdown SOPs + JSON structured data for equipment/patchbay/safety) and assembles them into a system prompt. Uses Vercel AI SDK (`ai` + `@ai-sdk/anthropic`) with `useChat()` on the client and `streamText()` on the server. Voice input via browser Web Speech API (free). No database, no RAG, no vector DB.

**Content layer:** Markdown SOPs and JSON structured data files in a `/content` directory with Zod-validated schemas. One directory per room (containing `overview.md`, `equipment.json`, `patchbay.json`, and `sops/*.md`), plus global `safety/rules.json` and `symptoms/categories.json`. Human-editable, git-versionable, trivially upgradable to a database later.

- Effort: S (1 weekend)
- Risk: Low
- Pros: Ships fast, zero infrastructure, voice/safety/symptom picker all work day one
- Cons: Doesn't scale past ~10 rooms, no analytics, no session persistence

### Approach B: "Supabase Full Stack"

Next.js PWA + Supabase backend with pgvector RAG, session tracking, analytics, symptom picker, staff escalation. The PRD's ideal architecture.

- Effort: M (2-3 weeks)
- Risk: Medium
- Pros: Scales to multi-studio, proper RAG, analytics from day one
- Cons: More infrastructure, longer to first demo, RAG accuracy needs tuning

### Approach C: "WhatsApp / SMS Bot"

Skip the web app. QR code opens a WhatsApp or SMS conversation with a bot. Voice messages, photos, and text work natively via the messaging platform.

- Effort: S-M (1-2 weeks)
- Risk: Medium
- Pros: Zero UI to build, native voice/photo, works on any phone
- Cons: No custom UI (no symptom picker, no visual safety warnings), platform dependency, per-message costs

## Recommended Approach

**Approach A: "Big Prompt, Small App"** with a markdown SOP + JSON structured data content layer.

Ship a working demo in a weekend. Real QR codes, real voice input, real safety guardrails, real symptom picker. Test with actual studio clients. Image upload deferred to v2. When usage outgrows the big-prompt approach (more rooms, need analytics, need admin SOP management), migrate to Approach B. The frontend barely changes: swap prompt-stuffing for a proper RAG pipeline.

### Architecture

```
Client (phone/laptop/studio computer)
  |
  | QR scan -> https://studio-assistant-one.vercel.app/room/room-a
  |
  v
Next.js API Route (Node.js runtime, app/api/chat/route.ts)
  |
  | 1. Rate limit via Vercel KV (50 req/hr per IP)
  | 2. Load room content (overview.md + equipment.json + patchbay.json + sops/*.md)
  | 3. Load safety/rules.json
  | 4. Deterministic safety keyword pre-scan on user message
  | 5. Assemble system prompt via lib/prompt.ts
  | 6. streamText() via Vercel AI SDK (@ai-sdk/anthropic)
  | 7. Stream response back via toUIMessageStreamResponse()
  |
  v
Claude API (claude-sonnet-4-20250514)
```

**Tech stack:** Next.js 16 + React 19 + Tailwind CSS 4 + Vercel AI SDK (`ai` + `@ai-sdk/anthropic` + `@ai-sdk/react`) + Zod 4 for content validation + `@vercel/kv` for rate limiting. Client uses `useChat()` hook; server uses `streamText()` with `convertToModelMessages()`.

**Model choice:** Uses `claude-sonnet-4-20250514` for all requests. Fast enough for streaming chat and smart enough for safety detection. Opus is overkill for this use case and would increase latency and cost. Revisit only if Sonnet demonstrably misses safety patterns during real-world testing.

**Runtime:** Node.js runtime (not Edge). This allows `fs.readFileSync` for loading content files from disk at request time, which simplifies the content layer.

**Hosting:** Deployed on Vercel at `studio-assistant-one.vercel.app`. Rate limiting via Vercel KV (50 requests per IP per hour, gracefully degrades when KV is not configured in dev).

### System Prompt Assembly

Implemented in `lib/prompt.ts`. The `assembleSystemPrompt(room, safetyRules)` function builds a system prompt from room content and safety rules:

1. **Role:** "You are the Autoland studio assistant for {room title}..." with goal of helping users set up sessions without staff intervention
2. **Room overview:** Full `overview.md` content (room description, dimensions, features)
3. **Equipment inventory:** Formatted from `equipment.json` — microphones (with type, weight, phantom requirements, location, notes), stands (with max weight), preamps (with inputs, phantom, patchbay position), and other gear
4. **Patchbay:** Layout description, default connections, and tips from `patchbay.json`
5. **SOPs:** All markdown SOP files from `sops/` directory, injected verbatim
6. **Safety rules:** All rules from `safety/rules.json` formatted as prominent warnings
7. **Behavioral instructions:** Jargon-free language, ask clarifying questions, step-by-step guidance, never assume hardware state, recommend "Get Staff Help" when unsure

When the deterministic safety pre-scan (in `lib/safety.ts`) detects a keyword match, a `[SAFETY RULE TRIGGERED]` block is appended to the system prompt instructing the model to address the warning prominently before any other guidance.

### Voice Input

Implemented in `components/VoiceInput.tsx`.

- **Engine:** Browser Web Speech API (free, runs client-side, works on Chrome/Safari/Edge)
- **Mode:** Continuous with interim results (`continuous: true`, `interimResults: true`)
- **Silence detection:** 2-second timeout after last speech result auto-stops recording
- **Max duration:** 15-second hard cap
- **Transcript handling:** Accumulated transcript is placed in the input textarea (not auto-sent), so the user can review and correct before sending
- **Visual feedback:** Pulsing red button with stop icon while recording; gray mic icon when idle
- **Unsupported browsers:** Mic button hidden entirely (falls back to text + symptom picker)
- **Fallback:** Text input and symptom picker always available
- **Future upgrade path:** Whisper API for better accuracy (server-side, costs money)

### Safety Guardrails

Safety rules are defined in `content/safety/rules.json` as JSON with keyword groups:

```json
{
  "rules": [
    {
      "id": "phantom-on-ribbon",
      "trigger": "phantom power on ribbon mic",
      "keywords": [["phantom", "ribbon"], ["48v", "ribbon"], ["phantom", "royer"]],
      "warning": "Engaging phantom power with a ribbon mic connected can permanently damage the mic. Only use phantom power with condenser microphones.",
      "severity": "critical"
    },
    {
      "id": "heavy-mic-light-stand",
      "trigger": "heavy mic on light stand",
      "keywords": [["tube", "small stand"], ["heavy", "light stand"], ["u87", "desk stand"]],
      "warning": "The tube LDC is heavy (~2.5 lbs). A lightweight stand can tip over and damage the mic. Please use one of the large tripod stands.",
      "severity": "high"
    }
  ]
}
```

**Enforcement strategy (two layers, both in MVP):**

1. **Deterministic server-side keyword pre-scan (`lib/safety.ts`):** Before forwarding the user message to Claude, the API route scans the last user message against all safety rules using `checkSafetyRules()`. Each rule has keyword groups (arrays of arrays) — a group matches if ALL of its keywords appear in the lowercased message. Rules are sorted by severity (critical > high > medium > low); the first matching rule wins. When a match is found, a `[SAFETY RULE TRIGGERED]` block is appended to the system prompt, forcing the model to address the warning prominently. This ensures critical safety warnings fire deterministically, independent of LLM behavior.

2. **Prompt-based:** The system prompt includes all safety rules with explicit instructions: "If the user describes any action matching a safety rule, STOP the conversation and issue the safety warning BEFORE providing any other guidance." This catches patterns the keyword scan might miss (e.g., paraphrased descriptions).

**Client-side detection:** The `isSafetyWarning()` function in the chat UI detects safety warnings in assistant messages (by checking for warning emoji, "SAFETY WARNING", or "Warning:" text) and renders them with distinct amber styling and a "Safety Warning" badge.

## SOP Creation Workflow

Nigel (studio owner) knows his rooms inside out but shouldn't need to learn JSON schemas or code. The actual workflow:

1. **Record:** Nigel does a Google call walking through each procedure — what gear is where, how the patchbay is wired, signal flow, what stands go with what mics, what clients screw up most often, what actions are dangerous. Gemini Note Taker automatically captures structured notes.

2. **Convert:** Feed Gemini's notes to Claude to convert into the content format: `overview.md` for the room description, `equipment.json` for structured equipment data (validated by Zod schemas), `patchbay.json` for wiring info, and individual markdown SOPs in `sops/`.

3. **Review:** Nigel reviews the generated content for accuracy. Did it get the mic weights right? Is the patchbay wiring correct? Quick pass, not writing from scratch.

4. **Commit:** Files go into `/content/rooms/{room-id}/`. Push to deploy.

This turns "write JSON and markdown files" (painful for a non-developer) into "walk through your room on a Google call" (natural for a studio engineer). Nigel's domain knowledge gets captured in his own words, then structured by AI.
