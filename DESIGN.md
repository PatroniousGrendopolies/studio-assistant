## Problem Statement

Autoland is a subscription-based recording studio where clients book rooms and use professional audio equipment. Onboarding currently takes 60-120 minutes of staff time per session. Clients range from total beginners (never touched a mic stand) to seasoned engineers. The biggest risks are equipment damage from misuse (heavy mics on light stands, phantom power on ribbon mics, bad cable handling) and client frustration when they can't figure out signal flow without help.

The goal: an AI chat assistant accessible via QR code in each room that cuts onboarding to under 30 minutes, catches dangerous actions before they happen, and lets clients get help without waiting for staff.

## What Makes This Cool

A studio client scans a QR code, says "I want to record vocals with the big tube mic," and the AI knows their room, knows that mic is heavy, asks what stand they're using, and catches them before they put a $5,000 mic on a $30 stand. That's the moment. Not a chatbot that answers FAQs. An assistant that understands analog studio physics and protects expensive gear through conversation.

The image troubleshooting is the other "whoa" moment: client uploads a photo of their patchbay, the AI sees that IN2 isn't patched to Preamp 2, and tells them exactly which cable to move.

## Constraints

- Mobile-first web app (PWA), no native app, no app store
- No login or accounts. QR scan to session. Zero friction.
- Cannot see or control hardware. All safety guardrails are triggered by what the user says, not by sensors or monitoring.
- Knowledge base must be human-reviewed. The AI cannot invent safety procedures.
- Must work on client phones, laptops, and studio computers.
- Budget-friendly: Vercel free tier + Claude API pay-per-use.
- SOPs must be updatable without code changes.

## Premises

1. **The knowledge base is the bottleneck, not the AI.** LLMs already know audio production well. The hard part is Autoland-specific room configs, wiring diagrams, and equipment inventories. These are lighter than a full audio engineering textbook, more like structured equipment lists and signal flow maps.

2. **Voice input and the symptom picker are essential for beginners, not nice-to-haves.** A client with no audio vocabulary won't type a useful query. The "I don't know what to say" button and voice input are the primary interface for most users.

3. **Safety guardrails are the #1 value driver.** One prevented equipment damage incident (phantom power on a ribbon mic, tube mic falling off a light stand) pays for the entire project. This drives every priority decision.

4. **No login for MVP.** QR code to session. Adding a login wall kills adoption in a studio where people are there to record, not to create accounts.

5. **The studio works today without this.** Staff onboarding works. This makes it faster and safer, but the MVP can be genuinely minimal because the fallback is "go find a staff member."

## Approaches Considered

### Approach A: "Big Prompt, Small App" (CHOSEN)

Simple Next.js or Vite app deployed on Vercel. QR code per room encodes a room ID in the URL. On load, the app reads structured YAML content files for that room (equipment inventory, wiring, safety rules) and assembles them into a system prompt. Calls Claude API from an edge function. Voice input via browser Web Speech API (free). Image upload via Claude's multimodal API. No database, no RAG, no vector DB.

**Content layer:** Structured YAML files in a `/content` directory with clear schema. One file per room, one for global safety rules, one for equipment catalog. Human-editable, git-versionable, trivially upgradable to a database later.

- Effort: S (1 weekend)
- Risk: Low
- Pros: Ships fast, zero infrastructure, voice/image/safety all work day one
- Cons: Doesn't scale past ~10 rooms, no analytics, no session persistence

### Approach B: "Supabase Full Stack"

Next.js PWA + Supabase backend with pgvector RAG, session tracking, analytics, symptom picker, staff escalation. The PRD's ideal architecture.

- Effort: M (2-3 weeks)
- Risk: Medium
- Pros: Scales to multi-studio, proper RAG, analytics from day one
- Cons: More infrastructure, longer to first demo, RAG accuracy needs tuning

### Approach C: "WhatsApp / SMS Bot"

Skip the web app. QR code opens a WhatsApp or SMS conversation with a bot. Voice messages, photos, and text work natively via the messaging platform.

- Effort: S-M (1-2 weeks)
- Risk: Medium
- Pros: Zero UI to build, native voice/photo, works on any phone
- Cons: No custom UI (no symptom picker, no visual safety warnings), platform dependency, per-message costs

## Recommended Approach

**Approach A: "Big Prompt, Small App"** with a structured YAML content layer.

Ship a working demo in a weekend. Real QR codes, real voice input, real safety guardrails, real image analysis. Test with actual studio clients. When usage outgrows the big-prompt approach (more rooms, need analytics, need admin SOP management), migrate to Approach B. The frontend barely changes: swap prompt-stuffing for a proper RAG pipeline.

### Architecture

```
Client (phone/laptop/studio computer)
  |
  | QR scan -> https://assistant.autoland.studio/room/a
  |
  v
Vercel Edge Function (API route)
  |
  | 1. Load room YAML + safety YAML + equipment YAML
  | 2. Assemble system prompt
  | 3. Forward chat messages + images to Claude API
  | 4. Stream response back
  |
  v
Claude API (claude-sonnet-4-6 for all requests)
```

**Model choice:** Use claude-sonnet-4-6 for all requests in MVP. It's fast enough for streaming chat and smart enough for safety detection and image analysis. Opus is overkill for this use case and would increase latency and cost. Revisit only if Sonnet demonstrably misses safety patterns during real-world testing.

**Hosting note:** Vercel's free tier (Hobby plan) has a 10-second execution limit on edge functions. The system prompt for a single room (room YAML + equipment catalog + safety rules) should fit within ~4,000-6,000 tokens of context. With Sonnet streaming, responses typically begin within 1-2 seconds and stream incrementally, which works within the timeout. If the assembled prompt grows beyond ~8,000 tokens (e.g., very large rooms with many equipment entries), consider Vercel Pro ($20/mo) or splitting the equipment catalog to only include items in the selected room. Test this on day one.

### Content Structure

```
/content
  /rooms
    room-a.yaml        # name, description, equipment list, patchbay wiring,
                        # signal flow, input/output mapping, special notes
    room-b.yaml
  /safety
    rules.yaml          # risky action patterns, warnings, escalation triggers
                        # e.g., "heavy mic + light stand" -> warning text
  /equipment
    catalog.yaml        # mic specs (weight, type, phantom requirements),
                        # stand ratings (max weight), preamp details,
                        # cable types, patchbay layout
  /symptoms
    categories.yaml     # symptom picker tiles: label, icon, follow-up questions
```

### System Prompt Assembly

For a given room, the edge function builds a system prompt:
1. Role: "You are the Autoland studio assistant for {room name}..."
2. Room context: equipment inventory, wiring, patchbay map from room YAML
3. Equipment specs: relevant entries from catalog (weights, phantom compatibility)
4. Safety rules: all risky action patterns and their warnings
5. Behavioral instructions: jargon-free language, ask clarifying questions, voice the safety warning prominently, never assume hardware state
6. Symptom handling: when user selects a symptom tile, inject that context into the conversation

### Key UI Components

1. **Room selector** (landing page after QR scan if room not in URL)
2. **Chat thread** with distinct bot/user/warning message styles
3. **Voice input** button (prominent mic icon, always visible)
4. **Photo upload** button (camera icon next to input)
5. **"I don't know what to say"** button that opens the symptom picker
6. **Symptom picker** grid with large, icon-based tiles
7. **Safety warning** cards with orange border, warning icon, specific guidance

### Voice Input

- Primary: Browser Web Speech API (free, runs client-side, works on Chrome/Safari/Edge)
- Fallback: Text input always available
- Future upgrade path: Whisper API for better accuracy (server-side, costs money)

**Degraded UX when voice fails (noisy studio, bad recognition):**
- Show the transcribed text in the input field BEFORE sending, so the user can review and correct it
- If the Web Speech API returns no result or very low confidence after 5 seconds, show a nudge: "Voice didn't catch that. Try typing, or tap a common problem below." and auto-open the symptom picker
- The mic button shows a visual recording indicator (pulsing red dot) so the user knows it's listening
- On browsers that don't support Web Speech API (rare, but Firefox desktop), hide the mic button entirely and rely on text + symptom picker

### Image Troubleshooting

- Client taps camera icon, takes/uploads a photo
- Photo sent to Claude's multimodal API alongside conversation context
- AI analyzes the image (patchbay connections, cable routing, equipment placement)
- Returns annotated guidance in text ("IN2 is not connected to Preamp 2. Connect the top jack to the bottom jack using a short patch cable.")

### Safety Guardrails

Implemented entirely in the system prompt. The safety rules YAML defines patterns:

```yaml
- trigger: "heavy mic on light stand"
  keywords: ["tube", "LDC", "heavy", "big mic", "small stand", "light stand", "black stand"]
  warning: "The tube LDC is heavy (~2.5 lbs). A lightweight stand can tip over and damage the mic. Please use one of the large tripod stands."
  severity: high
  action: recommend_alternative

- trigger: "phantom power on ribbon mic"
  keywords: ["phantom", "48v", "ribbon", "Royer", "Coles"]
  warning: "Engaging phantom power with a ribbon mic connected can permanently damage the mic. Only use phantom power with condenser microphones."
  severity: critical
  action: block_and_warn
```

**Enforcement strategy (two layers):**

1. **Prompt-based (MVP):** The system prompt includes all safety rules with explicit instructions: "If the user describes any action matching these patterns, STOP the normal conversation flow and issue the warning text BEFORE providing any other guidance. Use the exact warning text from the rules. Format it distinctly." This is the primary mechanism. It works because Claude is reliable at following safety-critical system prompt instructions, especially when the rules are explicit and the patterns are specific.

2. **Server-side keyword pre-scan (hardening, v1.5):** Before forwarding the user message to Claude, the edge function scans for high-severity keyword combinations (e.g., "phantom" + any ribbon mic name). If detected, the server injects a forced safety warning into the response stream immediately, independent of what Claude generates. This ensures critical safety warnings fire even if the LLM has an off day. Not needed for MVP launch but should be added after the first week of real-world testing confirms which patterns matter most.

**Accepted risk for MVP:** Prompt-only enforcement means there is a small chance Claude misses a safety pattern. Mitigated by: (a) the rules are specific and well-structured, (b) the PRD already states the assistant cannot replace staff for edge cases, (c) the studio has liability notices reminding clients to ask staff when unsure.

## Open Questions

1. **Which rooms to start with?** The friend should pick the 1-2 rooms with the most client issues. Start there, expand.
2. **Who writes the initial SOPs?** The friend (studio owner) writes the room YAMLs. LLM handles general audio knowledge. The YAML just needs room-specific facts.
3. **Should the assistant have a name/persona?** The PRD leaves this open. A warm, competent persona might help beginners trust it.
4. **Escalation to staff:** MVP can just show "Need a human? Ask the front desk." A webhook-based escalation (Slack/SMS) can be added in v2.
5. **Voice recognition accuracy in a studio:** Studios are noisy. Web Speech API might struggle with background music/instruments. Need to test in the actual environment.

## Voice Memo to YAML Workflow

The studio owner knows his rooms inside out but shouldn't need to learn YAML syntax. Instead:

1. **Record:** Friend opens Voice Memos on his phone and walks through each room, describing everything: what gear is where, how the patchbay is wired, signal flow, what stands go with what mics, what clients screw up most often, what actions are dangerous. 15-20 minutes per room, stream of consciousness. No structure needed.

2. **Transcribe:** Upload the voice memo to any transcription tool (iPhone can do it natively, or use Whisper, or paste into Claude). The raw transcript is the input.

3. **Generate:** Feed the transcript to Claude along with the room YAML schema (the example in this doc). Prompt: "Convert this studio engineer's description of their room into the following YAML schema. Ask me if anything is ambiguous." Claude outputs structured YAML.

4. **Review:** Friend reviews the generated YAML for accuracy. Did it get the mic weights right? Is the patchbay wiring correct? Quick pass, not writing from scratch.

5. **Commit:** The YAML file goes into `/content/rooms/`. Push to deploy.

This turns "write a YAML file" (painful for a non-developer) into "talk about your room for 15 minutes" (natural for a studio engineer). The friend's domain knowledge gets captured in his own words, then structured by AI.

**Prompt template for step 3:**

```
I'm going to give you a transcript of a studio engineer describing a recording
studio room. Convert it into a structured YAML file using this exact schema:

[paste the room YAML example from this design doc]

Rules:
- If the engineer mentions equipment, include all details: name, type, weight,
  location, and any safety notes
- If the engineer describes signal flow or patchbay wiring, capture it exactly
- If something is ambiguous or missing (e.g., mic weight not mentioned), flag it
  with a comment like "# TODO: confirm weight" so we can fill it in later
- Use the engineer's own words for location descriptions ("shelf by the window",
  "corner by the door") since clients will hear these same descriptions
```
